# KeepCalmAI Learning Experience Bible

## Purpose

KeepCalmAI exists to help people who feel overwhelmed, excluded or left behind by artificial intelligence become quietly confident through calm, accessible, practical learning experiences.

We do not teach AI through information overload. We build confidence through small, achievable successes.

## Core product statement

KeepCalmAI is the calmest way to learn AI.

The platform should feel less like a course library and more like a patient trainer sitting beside the learner.

## Learning experience principles

### 1. Calm before capability

People rarely stop learning because they lack intelligence. They stop because the experience feels too much. Every screen should reduce pressure before adding knowledge.

### 2. One screen, one idea

Each screen teaches one concept only. If a screen contains more than one learning objective, split it.

### 3. One action

Every screen should ask the learner to do one small thing. This may be reading, selecting, copying a prompt, asking a question, comparing answers or reflecting briefly.

### 4. One success

Every screen should end positively. Avoid failure language. If the learner is unsure, route them towards another explanation, not a judgement.

### 5. Scrolling is a warning

A learner should not need to scroll significantly to understand the point of a step. Scrolling usually means the content should be divided into smaller steps.

### 6. Progressive disclosure

Show only what the learner needs now. More detail can be expandable, optional or introduced later.

### 7. Practice quickly

Every concept should move into practice quickly. The learner should complete a meaningful AI interaction within the first five minutes of arriving.

### 8. Visible progress

Progress should be visible and reassuring. Use steps completed, resume cards and calm progress indicators rather than scores or grades.

### 9. Adapt quietly

A learner who chooses ChatGPT should see wording such as “Open ChatGPT”. A learner who chooses Microsoft Copilot should see “Open Microsoft Copilot”. The learning objective remains the same.

### 10. Accessibility is the default

Accessibility is not a mode. It is part of the product architecture. The platform must support keyboard use, screen reader use, low vision, cognitive accessibility, dyslexia, ADHD, autism, technology anxiety and older learners.

### 11. Confidence is the outcome

The learner should leave each journey thinking, “I can do this.” They do not need to remember everything.

### 12. End with one next step

Every journey should end with one achievable next action, not homework or a list of things to revise.

## Journey architecture

A journey is a guided learning experience made from micro-steps. A journey is not a long article.

Recommended structure:

1. Welcome
2. Orientation
3. First simple concept
4. First practical action
5. Reflection
6. Repeated learn, try and reflect steps
7. Safety or judgement step where relevant
8. Confidence check
9. Summary
10. Journey complete

## Step anatomy

Every step should include:

- title
- estimated time
- short body text
- why it matters
- one action title
- one small list of action steps
- optional choices
- optional example prompt
- ready to continue guidance
- key takeaway

## Screen types

Current approved screen types:

- Welcome
- Orientation
- Learn
- Try it
- Prompt
- Choice
- Reflection
- Safety
- Confidence check
- Summary
- Journey complete

New screen types should only be added when there is a clear learner need.

## Tone

The tone should be calm, direct and human. It should never feel patronising. Avoid hype, urgency and technical language unless it is explained simply.

Preferred language:

- “Let’s try one small step.”
- “You do not need to know everything.”
- “This is not a test.”
- “Continue when you feel ready.”

Avoid:

- “Master AI.”
- “Unlock your potential.”
- “Become an expert.”
- “Incorrect.”
- “You failed.”

## Product direction

The Journey Player is the core product pattern. Future journeys should be authored as structured data and rendered by reusable components wherever possible.
